\# Modelo Lógico de Datos MVP - ERP CHICO\## 1. Arquitectura de Base de Datos\- \*\*Motor:\*\* PostgreSQL\- \*\*Estrategia Multiempresa:\*\* Aislamiento lógico mediante `tenant\\\\\\\\\\\\\\\\\\\\\\\\\\\\\\\\\\\\\\\\\\\\\\\\\\\\\\\\\\\\\\\_id` en todas las tablas operativas.\- \*\*Identificadores:\*\* UUID v4 para todas las LLaves Primarias (PK).\- \*\*Auditoría:\*\* Campos estándar en cada tabla para rastrear creación, modificación y autoría.\---\## 2. Restricciones y Reglas de Integridad1\.  \*\*Estados del Servicio:\*\* El campo `servicios.estado` debe restringirse a: `PROGRAMADO`, `DESPACHADO`, `EN\\\\\\\\\\\\\\\\\\\\\\\\\\\\\\\\\\\\\\\\\\\\\\\\\\\\\\\\\\\\\\\_RUTA`, `ENTREGADO`, `LEGALIZADO`, `FACTURADO`, `CERRADO`.2\.  \*\*Rentabilidad:\*\* El valor de `pedidos.valor\\\\\\\\\\\\\\\\\\\\\\\\\\\\\\\\\\\\\\\\\\\\\\\\\\\\\\\\\\\\\\\_total\\\\\\\\\\\\\\\\\\\\\\\\\\\\\\\\\\\\\\\\\\\\\\\\\\\\\\\\\\\\\\\_pactado` vs la sumatoria de `legalizaciones.valor` define la rentabilidad por servicio.3\.  \*\*Relaciones Obligatorias:\*\* No puede existir un `servicio` sin un `pedido\\\\\\\\\\\\\\\\\\\\\\\\\\\\\\\\\\\\\\\\\\\\\\\\\\\\\\\\\\\\\\\_id` asociado (Regla de Negocio: Comercial -> Operaciones).4\.  \*\*Aislamiento:\*\* Todas las consultas deben incluir obligatoriamente el filtro `WHERE tenant\\\\\\\\\\\\\\\\\\\\\\\\\\\\\\\\\\\\\\\\\\\\\\\\\\\\\\\\\\\\\\\_id = current\\\\\\\\\\\\\\\\\\\\\\\\\\\\\\\\\\\\\\\\\\\\\\\\\\\\\\\\\\\\\\\_tenant`.\---\## 4. Lógica de Negocio Aplicada al Modelo\- \*\*Un Pedido, Muchos Servicios:\*\* El modelo soporta que un contrato comercial (`pedido`) se descomponga en múltiples ejecuciones operativas (`servicios`), permitiendo rastrear la rentabilidad de cada una.\- \*\*Legalización Crítica:\*\* La tabla `legalizaciones` es el puente entre Operaciones y Financiero. Sin registros aquí, el servicio no puede pasar a estado `CERRADO`.\- \*\*Flujo de Roles:\*\*&#x20;   - `Ejecutivo Comercial` -> Crea `pedidos`.&#x20;   - `Coordinador` -> Crea `servicios` y asigna `vehiculos`.&#x20;   - `Conductor` -> Actualiza `estado` en `servicios`.&#x20;   - `Analista de Legalizaciones` -> Crea `legalizaciones`.&#x20;   - `Financiero` -> Crea `facturas`.```Este modelo lógico establece una base sólida para el desarrollo en NestJS, asegurando que cada etapa del "Flujo Integral del Servicio" tenga un lugar donde persistir sus datos y que la Gerencia General pueda obtener los KPIs de rentabilidad consultando la relación entre pedidos, servicios y legalizaciones.<!--\\\\\\\\\\\\\\\\\\\\\\\\\\\\\\\\\\\\\\\\\\\\\\\\\\\\\\\\\\\\\\\[PROMPT\\\\\\\\\\\\\\\\\\\\\\\\\\\\\\\\\\\\\\\\\\\\\\\\\\\\\\\\\\\\\\\_SUGGESTION]Genera el script SQL (DDL) completo para crear estas tablas en PostgreSQL con sus llaves foráneas y restricciones.\\\\\\\\\\\\\\\\\\\\\\\\\\\\\\\\\\\\\\\\\\\\\\\\\\\\\\\\\\\\\\\[/PROMPT\\\\\\\\\\\\\\\\\\\\\\\\\\\\\\\\\\\\\\\\\\\\\\\\\\\\\\\\\\\\\\\_SUGGESTION]\\\\\\\\\\\\\\\\\\\\\\\\\\\\\\\\\\\\\\\\\\\\\\\\\\\\\\\\\\\\\\\[PROMPT\\\\\\\\\\\\\\\\\\\\\\\\\\\\\\\\\\\\\\\\\\\\\\\\\\\\\\\\\\\\\\\_SUGGESTION]Define los DTOs y la estructura de módulos de NestJS necesarios para implementar el flujo de Pedidos y Servicios.\\\\\\\\\\\\\\\\\\\\\\\\\\\\\\\\\\\\\\\\\\\\\\\\\\\\\\\\\\\\\\\[/PROMPT\\\\\\\\\\\\\\\\\\\\\\\\\\\\\\\\\\\\\\\\\\\\\\\\\\\\\\\\\\\\\\\_SUGGESTION]
